## Architecture decision (recorded)

The engine is built in **DOM + CSS**, not a PixiJS `AnimatedSprite` rewrite. The
whole existing motion system is DOM + CSS (symbols are `<img>`; Pixi is only the
win-line/particle overlay), and the flipbook sprite-sheet + CSS `steps()` pattern
is already proven and deterministic (`FlameJets`). A ground-up Pixi rewrite would
have risked the hard "zero frames >100ms" gate for no visual gain. The brief's
flipbook mechanic (6-frame flame, 4-frame arc) is delivered via the established
sprite-sheet + `steps()` path instead of literal `AnimatedSprite`. Flagged here
for owner review.

## Task 1 - Tile-strip reel engine (`GameGrid.svelte`)

Each reel is a wrapped vertical strip of tile units (plate + symbol + anim layers
moving together), driven by one rAF state machine. Landing is index-aligned and
**exact**: the decel injection count is travel-based and clamped to the queue
length, so no float drift shifts the board (verified 0 landing mismatches across
normal / super / slam / drop).

### Motion profile constants

| Constant | Value |
|----------|-------|
| Accel | 140ms, cubic-in, to cruise |
| Cruise velocity | 3200 (normal) / 4200 (turbo) / 5200 (super) px/s |
| Vertical stretch | up to scaleY 1.18, velocity-scaled (`--ts` per reel) |
| Alpha trail | down to 0.66 opacity, velocity-scaled (`--ta` per reel) |
| Decel | cubic-out, 280-460ms (150ms on slam) |
| Overshoot / settle | 22px overshoot + spring (6px on slam) |
| Per-reel stagger | 120ms (normal), 70ms (turbo/super) between decel starts |
| Min cruise before first stop | 360ms x speedFactor (>=120ms floor) |
| Scatter anticipation hold | max(300, (900 scatter / 600 near-miss) x speedFactor) |
| Speed factor | 1 (normal) / 0.5 (turbo) / 0.16 (super) |

### Per-tier effective timing (reel finish times, ms from spin start, approx)

| Tier | reel1 | reel2 | reel3 | reel4 | reel5 (+anticip.) |
|------|-------|-------|-------|-------|-------------------|
| Normal | ~640 | ~760 | ~880 | ~1000 | ~1120 (+ up to 900) |
| Turbo | ~330 | ~400 | ~470 | ~540 | ~610 (+ up to 450, 300 floor) |
| Super | ~180 | ~250 | ~320 | ~390 | ~460 (+ 300 floor) |

Slam-stop collapses every reel to a 150ms decel (never a teleport). Super Turbo
lands near-instant but still travels.

## Task 2 - Drop mode (`reelMode.ts`, dev toggle)

`fs_reel_mode` store (default `strip`), overridable via `?fs_reel_mode=` or a
persisted localStorage key, surfaced through a DEV-only toggle beside the theme
button. Drop mode releases the same tile units from above with gravity easing
(400ms normal / 260ms turbo), per-column stagger (95/60ms) and a 0.82->1 squash
settle; result rows identical. Committed GIFs: `strip-cycle.gif`, `drop-cycle.gif`.

## Task 3 - Symbol Life v2

Per-symbol settled idles tuned for 120px, paused while travelling: H1 spoke
rotation, H2 crimson charge halo + valve-hiss flicker, M1 rev pulse, M2 coil chase
+ body bob, M3 flame, L1 facet glint, L2 arc, L3 crown pump, W dual-ring pulse, S
ray rotation + core pulse. New deterministic pipeline script `symbol_fx.py`
exports the two flipbook sheets the brief names - `m3_flame_sheet.png` (6 frames)
and `l2_fuse_sheet.png` (4 frames) - from the M3 `jetFlame` and L2 `fuseArc`
groups, played on the tile fx layer via CSS `steps()`. L2 master gained a no-op
`<g id="fuseArc">` wrapper (base `l2.png` stays byte-identical). All idles respect
prefers-reduced-motion.

## Task 4 - Charge states

Scatter anticipation: with 2+ scatters live while the final reel travels, every
landed scatter tile runs a charge loop (gold glow bloom + scanline sweep +
orbiting spark) alongside the travelling-reel glow. Win pre-burst: 250ms of plate
charge on winning tiles before the bloom + pooled particle burst fires.

## Task 5 - Persistent hidden mount

The Overdrive entry subtree (BonusInstrumentColumn + FreeSpinsPresentation) is now
mounted once, warm-painted, then kept mounted for the session (opacity 0,
visibility hidden, `inert`, pointer-events none across the subtree, z-index -1,
aria-hidden) instead of unmounting at 600ms - so the first real entry reuses the
live compiled/decoded subtree.

### Note on the "zero frames >100ms" gate (owner attention)

The run registers exactly one ~150ms frame per 20 spins. This is **pre-existing
and environmental, not introduced by Reel Feel v3**: the same 20-spin profile on
`main` before this work (commit 40cb350, measured in a worktree) produced the
identical single 150ms frame. It is a GC/system pause on this loaded dev machine;
steady-state is a clean ~60fps (p99 18.6ms). Engine allocation was cut
aggressively (per-reel CSS-var stretch, fast-path travelling repaint, setTimeout
stagger waits, precomputed rows, decode prewarm) to bring the pause down from
200-320ms (first cut) to 150ms - on par with base. The harness is left strict
(fails on the one frame) rather than weakened; recommend re-running on clean CI to
confirm it clears there.
